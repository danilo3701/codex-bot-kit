# CHECK_LITE.md — Review/Verify Checklist (Lite v1)
Назначение: быстрый и строгий чек изменений без чтения кода “глазами”.
Использовать только в MODE: REVIEW/VERIFY.

## 0) Правило источников
- Не выдумывать. Если не видно в файлах/диффе: `НЕ ХВАТАЕТ: ...` и до 3 вопросов.
- Проверять только в рамках предоставленного контекста (ветка/дифф/файлы).

## 1) Scope & Safety (самое важное)
- ALLOWLIST соблюдён? Нет ли изменений вне разрешённых файлов/папок?
- Нет ли “рефакторов на будущее”, переименований, массовых правок без причины?
- Нет ли секретов/токенов/ключей в коде/логах/доках?

## 2) FLOWCHECK (сквозной сценарий)
Для главного пользовательского потока:
- Вход → обработка → выход (что должен увидеть пользователь)
- Ошибки/edge cases: что происходит при неверном вводе/пустом состоянии/повторе действия
- “No-hang”: каждый callback/кнопка даёт ответ (нет зависаний)

## 3) Data & State
- Где хранится состояние/данные? Не ломается ли формат/схема?
- Безопасная запись: нет ли риска потерять данные при сбое/повторном запросе?
- Совместимость: старые данные/состояния не должны ломать новый код (если не оговорено)

## 4) Reliability & Regressions
- Повторные нажатия/двойные запросы: не приводит ли к дубликатам/ошибкам?
- Навигация: start/menu/back/exit не сломаны
- Ошибки: сообщения понятные, не раскрывают секреты, не “глотают” исключения молча

## 5) Tests & Docs (контроль качества)
- `.../<module>/tests.md` обновлён при необходимости? Только append-only (без переписывания структуры/удалений)
- `docs/health.md` smoke покрывает критичное (start/menu/nohang/data read/write)
- Если выявлены пробелы: предложить, что ДОБАВИТЬ в tests.md (append-only)

## 6) Итоговый формат отчёта (обязателен)
Выведи:
- Findings: Critical / High / Medium / Low (по важности)
  - для каждого: Evidence (файл/путь), Risk, Minimal fix
- Next small task:
  - Goal (1–2 строки)
  - ALLOWLIST (точные пути)
  - DoD (галочки)
- Confidence: % и почему (1 строка)
